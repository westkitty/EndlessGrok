## Status Legend

- `planned`: identified but not specified.
- `prompted`: production handoff/spec exists.
- `generated-unverified`: file exists but QA is pending.
- `generated-verified`: file exists and visual/functional QA passed.
- `implemented`: integrated into project code/data.
- `needs-revision`: failed QA or canon check.
- `deferred`: intentionally postponed.

## Canonical asset ID scheme

Runtime and manifest rows use hyphenated, family-prefixed IDs:

| Family | Pattern | Example |
|---|---|---|
| resources | `resource-{mechanical-key}` | `resource-starsilk-thread` |
| victory | `victory-{mechanical-key}` | `victory-starbinding` |
| macros | `macro-{mechanical-key}` | `macro-syrin-inerting-mist` |
| macro effects | `macro-effect-{mechanical-key}` | `macro-effect-syrin-inerting-mist` |
| map states | `map-state-{mechanical-key}` | `map-state-collapsed` |
| deposits | `deposit-{mechanical-key}` | `deposit-blood-ring-glass` |
| factions | `faction-{mechanical-key}` | `faction-ledger-administration-emblem` |
| fleet roles | `fleet-{mechanical-key}` | `fleet-scout-icon` |
| ship modules | `ship-module-{mechanical-key}` | `module-weapon-star-knife-lance` |
| events | `event-{mechanical-key}` | `event-victory-heliocide-confirmation` |

Mechanical keys (`resource:starsilkThread`, `macro:syrin_inerting_mist`) are stable game identifiers and do not change during reconciliation.

## Legacy ID alias policy

Older registry IDs (e.g. `starsilk-thread`, `macro-syrin_inerting_mist`) resolve to canonical IDs through `src/data/assets/assetIdReconciliation.ts`. The registry indexes both legacy and canonical IDs so existing consumers keep working.

Helpers:

- `getCanonicalAssetId` — map legacy → canonical
- `resolveAssetId` — resolve mechanical key, legacy, or canonical ID
- `getAssetIdAliases` — list all known aliases for an asset
- `detectAssetIdConflicts` / `assertNoDuplicateMechanicalKeys` — drift checks

Canonical IDs are preferred for new code. Legacy aliases are temporary compatibility, not a second source of truth.

## Manifest runtime lookup

Two layers coexist:

1. **Integrated registry** (`src/data/assets/records/*.ts`) — assets with icon/tooltip contracts wired into UI today.
2. **Ledger manifest** (`src/data/assets/assets.manifest.json`) — full 134-row Starsilk ledger, all `prompted`.

UI resolution path:

```
AssetIcon / ResourceBar / VictoryPanel
  → resolve.ts (getAssetIconName, buildRuntimeTooltip)
    → registry.ts (integrated records)
    → runtimeResolve.ts → runtimeManifest.ts (getManifestAssetById)
```

Import `getManifestAssetById` from `src/data/assets/index.ts` (aliased export of `getAssetById` from `runtimeManifest.ts`).

## UI consumption rules

- Use mechanical keys in components (`resource:starsilkThread`, `victory:starbinding`).
- `AssetIcon` resolves through `resolve.ts`; missing records fall back to family-default icons with stable `data-testid`.
- Tooltips use `buildRuntimeTooltip()` — mechanics first, lore second, warning third.
- Do not mark a row `integrated` or `generated-verified` until the referenced file exists on disk and validation passes.

## Future generated asset paths

Production SVG/audio land under:

- `public/assets/icons/{resources,victory,macros,factions,fleets,ui}/`
- `public/assets/audio/`

Test-only fixture SVGs belong under `tests/fixtures/assets/` and must never be confused with production assets.

## Upstream package boundary

The Starsilk Asset Production Agent lives externally at `starsilk_4x_assets/starsilk_asset_agent_full_context/`. Do not copy or vendor that package into `src/`. Ingest generated batches through manifest validation (`npm run assets:validate`) and registry merge (`integration.ts`), then update ledger rows from `prompted` → `generated-unverified` only when files exist.
